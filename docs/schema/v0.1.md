# `ReportV01` schema ŌĆö consumer compatibility guide

The whatif report's wire format is `ReportV01`, declared in [`src/whatif/report/models_v01.py`](../../src/whatif/report/models_v01.py) and rendered to JSON Schema at [`src/whatif/report/schema/v0.1.schema.json`](../../src/whatif/report/schema/v0.1.schema.json). This page documents what consumers can rely on across v0.1.x patch releases and what changes are reserved for v0.2.

## Stability contract

**v0.1.x patches will NOT change:**

- Field names of the twelve top-level keys (`schema_version`, `schema_uri`, `verdict`, `verdict_state`, `cohort_results`, `failures`, `decision_findings`, `cache_summary`, `trust_floor`, `decision_policy`, `methodology`, `runtime`).
- The closed `verdict_state` literal set: `"ship"` | `"dont_ship"` | `"inconclusive"`.
- The semantics of every currently-required field.
- `schema_version` is fixed at `"v0.1"` for the entire v0.1.x line.

**v0.1.x patches MAY:**

- Add new optional fields. Consumers MUST tolerate unknown keys (cardinal #6 typed-extension-point discipline).
- Add new entries to the `failure_codes` and `finding_codes` registries (extension points).
- Refine human-readable messages inside `failures[].message`, `decision_findings[].rationale`, etc. ŌĆö these are intentionally not load-bearing for programmatic consumers.

**v0.2 may break:**

- `schema_version` bumps to `"v0.2"`.
- Field renames, removals, or semantic shifts will be called out under `### Changed (BREAKING)` in [CHANGELOG.md](../../CHANGELOG.md).
- `whatif report-migrate` (currently a v0.1 no-op stub) will gain real migration logic when v0.2 ships.

## Top-level shape

```json
{
  "schema_version": "v0.1",
  "schema_uri": "https://whatif.codes/schema/report/v0.1.json",
  "verdict": { ... },
  "verdict_state": "ship" | "dont_ship" | "inconclusive",
  "cohort_results": [ ... ],
  "failures": [ ... ],
  "decision_findings": [ ... ],
  "cache_summary": { ... },
  "trust_floor": { ... },
  "decision_policy": { ... },
  "methodology": { ... },
  "runtime": { ... }
}
```

## `verdict_state` and exit code

The string at the top level is the canonical machine-readable verdict. Consumers should branch on this, NOT parse `verdict.kind` or the rendered Markdown.

| `verdict_state` | CLI exit code | Meaning |
|---|---|---|
| `"ship"` | `0` | Floor passed; no above-floor `blocks_ship` finding. Construction requires a `FloorPassedProof` witness token (cardinal #2). |
| `"dont_ship"` | `1` | Floor passed; at least one above-floor guard emitted a `blocks_ship` finding. |
| `"inconclusive"` | `2` | Floor failed (cardinal #2 ŌĆö overrides policy) OR setup failure prevented producing a verdict. |

CLI exit codes also overload `2` for setup failures (config error, missing adapter credentials, malformed runner-target reference). Pipeline-internal code distinguishes via `report.verdict_state`; the CLI shell collapses both into exit `2` per `src/whatif/cli.py:18-29`.

## Determinism subset (cardinal #4)

Each field in the JSON Schema carries an `x-deterministic: true|false` annotation. Fields marked deterministic MUST produce byte-identical output across runs of the same config + seed. Examples:

- `runtime.config_hash` ŌĆö sha256 of the canonical config JSON
- `runtime.selection_seed`
- `trust_floor.*`, `decision_policy.*`
- `methodology.bootstrap.{method,resamples,seed,sample_unit,ci_level}`

Non-deterministic fields (clearly disclosed):

- `runtime.started_at`, `finished_at`, `duration_ms`
- `runtime.environment.python`, `platform`
- `failures[].message` (human prose)

Consumers building deterministic regression tests should extract the deterministic subset via `whatif.serialization.determinism.extract_deterministic_subset_from_report`. The integration test `tests/integration/test_determinism.py` is the load-bearing reference for byte-equality.

## Methodology disclosure (cardinal #10)

`methodology` is REQUIRED on every report. Consumers MUST NOT treat absence as "no methodology was used."

Sub-disclosures:

- `methodology.unit_of_analysis` ŌĆö always `"paired_trace_delta"` in v0.1
- `methodology.primary_metric`, `primary_endpoints`, `cohorts` ŌĆö predeclared at run start
- `methodology.bootstrap` ŌĆö bootstrap method + parameters. v0.1 ships `method="unavailable"` with `unavailable_reason="v0.1 empirical-percentile shortcut..."` truthfully (see [`phases.md`](../../.claude/skills/whatif-design/references/phases.md#disclosed-shortcuts-not-release-blockers-truthfully-declared-in-the-report)). v0.2 stats layer flips this to `"paired_percentile_bootstrap"`.
- `methodology.multiplicity` ŌĆö `correction` + `reason` for primary-endpoint multiplicity adjustment
- `methodology.judge` ŌĆö full judge-side disclosure: `scorer`, `scorer_version`, `judge_provider`, `judge_model`, prompt + rubric hashes, cache stats, the **five reproducibility concepts** (`reproducibility_addressed`, `reliability_measured`, `validity_measured`, `calibration_measured`, `bias_audit_measured`)
- `methodology.effect_size` ŌĆö `practical_delta`, source, judge noise floor
- `methodology.per_trace_inference` ŌĆö always `"descriptive_only"` in v0.1 (per-trace deltas are descriptive evidence; primary cohort endpoints drive the verdict)
- `methodology.causal_claim_scope` ŌĆö always `"associated_under_cached_tool_replay"` in v0.1

## Failure codes registry

`failures[].code` references entries in the `FAILURE_CODE_REGISTRY` ([`src/whatif/decision/failure_codes.py`](../../src/whatif/decision/failure_codes.py)). v0.1 codes:

| Code | Stage | Scope | Retryable |
|---|---|---|---|
| `runner_timeout` | replay | trace | true |
| `runner_exception` | replay | trace | false |
| `tool_cache_miss` | replay | trace | false |
| `scorer_unavailable` | score | trace | true |
| `scorer_response_malformed` | score | trace | false |
| `trace_schema_mismatch` | ingest | trace | false |
| `replay_validity_below_floor` | replay | cohort | false |
| `min_replayed_per_required_cohort` | floor | cohort | false |
| `min_scored_per_required_cohort` | floor | cohort | false |
| `min_replay_validity_ratio_per_required_cohort` | floor | cohort | false |
| `cache_lock_unavailable` | setup | run | true |
| `selection_failed` | setup | run | false |

The registry is the authoritative list ŌĆö consumers should lookup against it rather than hardcoding strings.

`failures[].details` is `Mapping[str, JsonPrimitive]` ŌĆö a typed extension-point map per cardinal #6. Specific codes set specific keys (e.g., `runner_timeout` carries `timeout_seconds`; `scorer_unavailable` carries `provider`, `reason`, `exc_type`, optionally `replay_code` or `rationale_classification` when the cli_pipeline closure raised).

## Finding codes registry

`decision_findings[].code` references entries in `FINDING_CODE_REGISTRY` ([`src/whatif/decision/finding_codes.py`](../../src/whatif/decision/finding_codes.py)). Severity ladder: `info` < `warning` < `blocks_ship` < `blocks_all`.

Blocking findings (`blocks_ship` or `blocks_all`) MUST have a registered fix-suggestion template in `FIX_SUGGESTION_REGISTRY` ([`src/whatif/decision/fix_suggestions.py`](../../src/whatif/decision/fix_suggestions.py)) so the rendered Markdown produces actionable next steps (cardinal #8).

## Cohort results

`cohort_results[]` is the per-cohort summary. Each entry has:

- `name`, `selected`, `replayed`, `scored`
- `ci_computable`, `ci_unavailable_reason` (closed enum: `"sample_too_small"` | `"floor_failed"` | `"insufficient_variance"`)
- `median_delta`, `ci_lower`, `ci_upper` ŌĆö `DecimalString` ("0.123" format) when `ci_computable`, else `null`
- `floor_passed`, `floor_failures` ŌĆö cardinal-#2 surface
- `improved_count`, `unchanged_count`, `regressed_count` ŌĆö per `practical_delta_epsilon`
- `ci_meaningful` ŌĆö Boolean derived from `ci_computable` AND `ci_lower`/`ci_upper` width

## Schema validation

The committed schema at `src/whatif/report/schema/v0.1.schema.json` is byte-stable across v0.1.x. The CI determinism check runs:

```
diff src/whatif/report/schema/v0.1.schema.json <(uv run python scripts/generate_schema.py --stdout)
```

so any internal-type refactor that would mutate the wire format fails the gate, even if every test still passes (cardinal #6 hand-written-schema discipline).

## Migration

v0.1.x ŌåÆ v0.1.x: byte-stable, no migration needed.

v0.1.x ŌåÆ v0.2: `whatif report-migrate <report.json>` will provide structured migration. Currently a no-op returning `EXIT_SUCCESS` for any v0.1.x input (no fields removed within v0.1.x; intentional no-op rather than setup-failure exit code so operators wiring this into automated pipelines aren't misled).

## Reading a report programmatically

```python
import json
from pathlib import Path

report = json.loads(Path("./reports/whatif-fork-2026-05-09.json").read_text())

assert report["schema_version"] == "v0.1"

if report["verdict_state"] == "ship":
    print("Ship ŌĆö verdict carries FloorPassedProof structurally.")
elif report["verdict_state"] == "dont_ship":
    blocking = [f for f in report["decision_findings"] if f["severity"] in ("blocks_ship", "blocks_all")]
    for finding in blocking:
        print(f"Don't ship: {finding['code']} ŌĆö {finding['rationale']}")
        for step in finding.get("fix_suggestion", {}).get("steps", []):
            print(f"  - {step}")
else:  # inconclusive
    floor_failures = [
        ff for cohort in report["cohort_results"]
        for ff in cohort["floor_failures"]
    ]
    print(f"Inconclusive: {len(floor_failures)} floor failure(s)")
```
