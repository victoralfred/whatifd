### Pipeline 2: Adapter projection → RawTrace ingress (cardinal-#5 boundary)

**Entry (per adapter):**
- `whatifd-langfuse`: `LangfuseTraceSource.iter_traces()` (`packages/whatifd-langfuse/src/whatifd_langfuse/source.py:139`)
- `whatifd-phoenix`: `PhoenixTraceSource.iter_traces()` (`packages/whatifd-phoenix/src/whatifd_phoenix/source.py:163`)
- `whatifd-inspect-ai`: `InspectAIScorer.score(case)` (`packages/whatifd-inspect-ai/src/whatifd_inspect_ai/scorer.py:124`)

**Steps (combined cross-adapter):**

1. **External-source ingestion** — Langfuse: paginated `api.trace.list(page=N, limit=...)` (line 142-156). Phoenix: caller-supplied `spans_provider()` materializes spans into a dict (line 164-184). Inspect AI: caller-supplied `score_fn(case)` invocation, with `asyncio.run()` for async return (line 125-132).
2. **PII-attribute wrapping (TraceSource adapters)** — both Langfuse and Phoenix pipe metadata through `whatifd.adapters.pii.wrap_pii_attributes(...)` at the projection boundary (`langfuse/source.py:209`, `phoenix/source.py:264`). Inspect AI is a Scorer; no metadata-PII surface.
3. **User-content `Sensitive[str]` wrapping** — Langfuse wraps `input` and `output` at `_project` (line 199-200). Phoenix pre-wraps `input.value` and `output.value` on every span via `_wrap_user_content_in_span` (line 84-110) before the classifier sees them. Inspect AI wraps `explanation` at `_project_score` (line 211).
4. **Cohort classification** — Langfuse and Phoenix invoke caller-supplied `cohort_classifier(trace_or_spans)` (langfuse line 198, phoenix line 255). Inspect AI is per-case; no cohort classification.
5. **Pydantic validation boundary** — `RawTrace(...)` construction triggers `_enforce_pii_attribute_wrapping` model_validator (`src/whatifd/adapters/protocols.py:175-212`), which raises `PIIAttributeTypeError` if any `PII_ATTRIBUTE_KEYS` member is unwrapped. `JudgeResult(...)` construction validates the Sensitive[str] rationale.
6. **Yield/return** — TraceSources yield `RawTrace` lazily into `run_pipeline`. Scorer returns `JudgeResult` to the `delta_fn` caller.

**Terminal state:** `RawTrace` / `JudgeResult` instances flow into the core pipeline; the cardinal-#5 wrapping is structurally enforced at construction.

### Pipeline 3: Cache lock lifecycle

**Entry:** `acquire_cache_lock(cache_root, *, stale_after_seconds, allow_age_takeover)` (`src/whatifd/cache/lock.py:229-350`)

**Steps:**

1. **Module-load guard** — `cache/lock.py:125-131` raises `ImportError` on Windows with an actionable typed message (cardinal #1 at import time).
2. **fd creation** — `os.open(lock_path, O_RDWR | O_CREAT, 0o600)` + `os.fdopen` (line 273-274); cache root mkdir at line 255.
3. **Initial `fcntl.flock(LOCK_EX | LOCK_NB)`** — line 277-279.
4. **On `BlockingIOError`**: `_try_takeover_if_stale` (line 353-377) reads the existing lock file, deserializes via `parse_lock_file_content`, and calls `_should_takeover` (pure decision: PID-dead OR PID-recycled OR age-exceeded-with-opt-in).
5. **Stale takeover** — `_process_dead_or_recycled` (line 398-414) uses `psutil.Process(pid).create_time()` with a 1.0-second tolerance; `AccessDenied` is "conservative don't-take-over" (line 408-412).
6. **Re-flock after takeover** (line 288-294); if the re-flock still fails, raise `_build_locked_error`.
7. **`OSError` ENOLCK/EOPNOTSUPP branch** — line 295-305; documents NFS limitation as DATA condition.

9. **Yield `CacheLock(lock_path, content)`** (line 314).
10. **Cleanup** (line 315-350): `LOCK_UN` → `close(fd)` → `unlink(lock_path)`. All three steps wrapped to surface failures via `ResourceWarning` without masking the original exception.

### Executive summary

whatifd v0.2.0 has shipped four packages to PyPI with ~1316 tests green and is **broadly production-acceptable for the v0.1 fork CLI happy path with cache `mode="off"`** (the shipping default). Eight components are **production-hardened today** — the cardinal-rule enforcement primitives (Sensitive[T] three-layer chain, FloorPassedProof witness, decision-pipeline structural integrity, serialization graph walk, statistical layer) are intact and reviewed-no-concerns across all five pipelines. The design's load-bearing invariants hold.

**Three components are not production-ready** and need fixes before any expanded-scope deployment: (1) `whatifd.adapters.factory` (F-1.1, one-line docs-vs-code contract break against the Phoenix adapter that already ships), (2) `whatifd.cache` lock (F-3.1, fcntl/unlink race defeats single-writer guarantee under concurrent fork — reproducible on multi-tenant CI runners), and (3) `whatifd.cli` dispatcher (F-1.3 stack-trace leak past cardinal-#1 boundary + F-1.6 methodology mislabel — compounding P1s). All three are small-effort fixes (S effort each) and should ship as a v0.2.1 hardening release **before** v0.3 work begins; they are independent of every v0.3 gating decision.

**`whatifd-inspect-ai` bifurcates by configuration:** acceptable with caveats when cache is off (the default), not production-ready when cache is on (F-2.1 silent-wrong-verdict). The fix lives at the shared `CacheKeyComponents` schema, not per-adapter — adding an `output_hash` field there closes the defect across both shipped scorer adapters.

**Top three risks in order of impact:**

1. **Wire-contract durability has no enforcement** (F-5.1 + F-5.4 + F-5.2). The schema URLs at `whatif.codes/schema/report/v0.X.json` are permanent contracts stamped into every emitted report, but nothing prevents the docs site from drifting from library releases, deleting retired versions, or — via the `models_v01.py`/v0.2-content confusion — letting a contributor mis-edit the wire shape. None of this is broken today; all of it is a time-bomb as v0.3 schemas ship. Address before v0.3's third schema lands.
2. **Bounded-memory contract is broken end-to-end** (F-4.1 + F-2.4). The Phase 6 plan gate claims streaming bounds, but `list(trace_source.iter_traces())` + Phoenix's full-span materialization mean peak memory is O(N) regardless of adapter laziness. No current test catches it because no test runs at scale. This is the v0.3 architectural decision that gates the most other work.
3. **The two P1 concurrency/integrity defects** (F-3.1 flock/unlink race + F-2.1 cache-key omits outputs). Both are reproducible failure modes in plausible deployments (multi-tenant CI; programmatic users enabling scorer cache). Both have small-to-medium fixes with clear surfaces. Neither requires a gating decision.

**Recommended remediation order:**

1. **v0.2.1 hardening** — ship F-1.1, F-1.3, F-3.1, F-2.1, F-5.3, F-5.5, F-4.7, F-4.10. All trivial-to-small effort, no gating decisions. Closes the only two "not production-ready" component verdicts.
2. **v0.3 scoping doc decides (a)–(g)** — bounded-memory contract, scorer-metadata threading, failure-code registry policy, Sensitive-in-exception convention, docs-CI parity, module rename strategy, PINNED.md adoption.
3. **v0.3 implementation** — F-4.1/F-2.4 streaming, F-1.6/F-4.2 metadata threading, F-4.5/F-4.6/F-4.9 registry expansion, F-4.3 sanitizer, F-5.1/F-5.2/F-5.4 wire-contract durability.

**Overall production posture:** whatifd is closer to production-hardened than the 5 P1 findings suggest. The cardinal-rule enforcement is structurally sound; the P1s are concrete, locally-fixable defects rather than design-level concerns. A v0.2.1 hardening release closes the not-production-ready verdicts; the v0.3 work is about durability and scale, not correctness.
